## 2. 調査結果

### 2.1 ベースイメージを更新しないことのリスク

#### 2.1.1 セキュリティ的なリスク — **仮定は正しい**

ベースイメージを更新しないことの**セキュリティリスクは明確に存在する**。一次資料（OpenTofu Issue #1931）でも「安全なベースイメージとするには少なくとも週次でパッケージ更新を出し続ける必要がある」とされており、調査結果とも一致する。

- **脆弱性の蓄積**: Chainguard の分析では、ベースイメージのバージョンを放置すると、脆弱性数が**約 200〜2500% 増加**する事例がある（NGINX:alpine の 1.18.0 → 1.20.2 未更新で、数個から 30〜50 個へなど）。
- **EOL イメージ**: サポート終了したベースイメージは、半年あたり平均 **218 件の脆弱性**が蓄積するという調査があり、その多くはアプリ本体ではなく**イメージ内コンポーネント**に起因する。
- **既知の CVE 例**: OpenSSL（CVE-2022-3602, CVE-2022-3786）、curl（CVE-2023-38545, CVE-2023-38546）など、古いベースイメージに含まれるライブラリに影響する重大な CVE が多数報告されている。
- **攻撃面の拡大**: 古いイメージにはビルドツール・パッケージキャッシュ・デバッグ用バイナリが残りやすく、攻撃者が侵入した後の横移動・権限昇格に利用されうる。

**結論**: 「セキュリティ的なリスク」という仮定は**正しい**。

#### 2.1.2 CA 証明書の outdated — **仮定は正しい**

古いベースイメージのままにしておくことで、**CA 証明書の期限切れ・不足**により SSL/TLS 検証が失敗するリスクがある。

- 古い Ubuntu イメージなどでは、**ルート CA（例: DST Root CA）の期限切れ**により、`wget` や `curl` が「certificate has expired」で失敗する事例が報告されている。
- Let's Encrypt など現代の証明書チェーンには、**OpenSSL 1.1.x 以降**など、ある程度新しい環境が必要になる場合がある。
- コンテナはイミュータブルなため、**証明書の更新にはイメージの再ビルド・再デプロイ**が必要で、ベースを更新しない限り古い証明書ストアのままになる。

**結論**: 「CA 証明書の outdated」という仮定も**正しい**。Terraform v1.10 では Alpine のデフォルト変更で `update-ca-certificates` がイメージから消え、ユーザーから問題が報告された事例もある。

#### 2.1.3 その他に考えられるリスク

- **フリート全体への影響**: 1 つの脆弱なベースイメージが、その上にビルドされた多数のサービスにそのまま継承され、「 blast radius 」が広がる。
- **サイレントな脆弱性の増加**: 上流のベースイメージが**タグを変えずにセキュリティ修正でリビルド**することがあり、古いタグにピン留めしているとその修正を取り込めない。
- **修正の複雑さ**: コンテナのイミュータブル性のため、修正には**フルリビルド・再デプロイ**と、新規 OS パッチとの互換性確認が必要で、更新を先送りしがちになる。
- **Alpine の事例**: 古い Alpine の安定版では、CVE 修正が main にのみ入り安定版にバックポートされず、ユーザーが Dockerfile 内でパッケージを更新しても依存関係で解決できず、公式イメージのリビルドを待つしかない場合がある。

### 2.2 他 OSS のベースイメージ更新（特に Terraform）

#### 2.2.1 Terraform（HashiCorp）のポリシー

- **ベースイメージ**: 公式 Terraform Docker イメージは **Alpine Linux** をベースとしており、リポジトリの Dockerfile では **`alpine:latest`** を利用している（2020 年頃から `alpine:latest` で統一）。
- **ビルド方式**: 公式イメージは **リリース用バイナリをそのまま含む**形態で、ソースからコンテナ内でビルドしているわけではない。Dockerfile はリポジトリにあり、公式リリース用ビルドは別プロセス。
- **バージョンタグ**: ユーザー向けには **明示的なバージョンタグ**（例: `1.14.6`）の利用が推奨されている。Terraform Enterprise では「明示的バージョンのイメージを使用すること」が公式に案内されている。
- **古いアプリバージョンへのベース更新**: 公開情報からは、**過去の Terraform バージョン用イメージについて、ベースイメージ（Alpine）を定期的に更新している様子は見当たらない**。新バージョンリリース時にその時点の `alpine:latest` でビルドされるという形が自然であり、「古い 1.x 用イメージを毎月ベースだけ更新して再ビルドする」ような運用は説明されていない。**「おそらく行っていない」という想定は妥当**と考えられる。
- **最近の事例**: Terraform v1.10 で、Alpine のデフォルトパッケージ構成変更により `update-ca-certificates` が含まれなくなり、ユーザーから問題が報告されている。Vault や Packer では以前から `ca-certificates` を明示インストールしているとの指摘があり、Terraform 側の今後の対応が議論されている。→ [Terraform Docker Image v1.10 does not contain update-ca-certificates anymore (Issue #36431)](https://github.com/hashicorp/terraform/issues/36431)

#### 2.2.2 新規リリース時のベースイメージ更新と Dependabot

- **新規アプリバージョンリリース時**: 多くの OSS では、**新バージョンをリリースするタイミングで、その時点の Dockerfile に書かれたベース（例: `alpine:latest`）でイメージをビルドする**ため、「そのリリース時点」のベースにはなる。ただし、**リリース後にベースイメージ側だけが更新されても、既存のアプリバージョン用イメージが自動でリビルドされる仕組みは、Terraform の公開情報からは見当たらない**。
- **Dependabot**: GitHub の Dependabot は **Docker ベースイメージの更新検知と PR 作成**に対応している。`dependabot.yml` で `package-ecosystem: "docker"` を指定し、`directory: "/"` などで Dockerfile を指定すると、ベースイメージの新バージョンが出た際に **PR が自動で開かれる**。これにより「ベースイメージのバージョンアップ」は自動化できるが、**その PR をマージしたうえで、どのタイミングでどのアプリバージョン用イメージをリビルドするか**はリリースプロセス次第。Dependabot は「ベースの更新 PR を出す」までで、**既存タグのイメージの再ビルド・再プッシュ**は行わない。
- **類似の機能**: 他の CI（Renovate 等）でも Docker ベースイメージの更新検知は可能。いずれも「Dockerfile の FROM を更新する PR」であり、**タグ付き既存イメージの定期的なリビルド**は別途、リリースパイプラインや Cron ジョブで設計する必要がある。

#### 2.2.3 ベースイメージを自動更新することのリスクと軽減

- **リスク**: ベースイメージのバージョンを自動的に上げると、**破壊的変更や挙動差**でビルド・実行が失敗したり、本番に影響したりする可能性がある（例: Alpine のパッケージ構成変更で `ca-certificates` が消える等）。
- **軽減策**:
  - **CI での自動テスト**: ベース更新 PR に対して、ビルド・単体・結合テストを必ず実行し、失敗したらマージしない。
  - **ステージングでの検証**: 本番前にステージング環境で動作確認する。
  - **カナリアリリース・ロールバック**: 問題が出た場合に影響範囲を限定し、ロールバックしやすくする。
  - **キュレーションされたレジストリ**: 上流のベースイメージをそのまま使わず、脆弱性スキャンや検証を通したうえで自前のレジストリに取り込んでから使う方法。
  - **小さく頻繁に更新**: 大きなジャンプより、小さい変更を頻繁に当てることで、原因切り分けをしやすくする。

Chainguard のブログでは「更新を避けると脆弱性が蓄積する一方、更新には破壊のリスクがある。だからこそ、**テストを充実させたうえで、むしろ継続的に（高頻度で）更新する**ことが推奨されている」とまとめられている。

### 2.3 Alpine から distroless に変えた場合の性質の変化

**問題の「性質」は変わる。維持の負荷とリスクの種類が変わる。**

- **Alpine**:
  - 通常の Linux ディストリビューションとして **パッケージ（musl、openssl、ca-certificates 等）の継続的なパッチ適用**が必要。
  - シェル・パッケージマネージャ（apk）・各種ユーティリティが含まれるため、**攻撃面と CVE の対象**が多くなる。
  - ベースイメージを更新しない場合、**セキュリティリスクと CA 証明書の outdated の両方**が、そのまま残りやすい。

- **Distroless**:
  - **シェル・パッケージマネージャ・多くの OS ツールを含まない**ため、パッチを当てる「OS」がほぼ存在しない形になる。
  - **CVE の多くは「パッケージを更新する」対象が減る**ため、**パッチ運用の負荷は小さくなる**。攻撃面も減る。
  - 含まれるのは主に **ランタイム依存（例: libc）とアプリバイナリ**であり、CA 証明書などをどう含めるかはイメージ設計次第（distroless でも CA バンドルは含められる）。
  - **デバッグが難しくなる**（シェルがないため `docker exec` で入りにくい）ので、Docker Debug やエフェメラルなデバッグ用コンテナなどの運用が前提になる。

**結論**:  
- **「ベースイメージのパッケージを継続的に更新する」という工数・リスクは、distroless にするとかなり減る。**  
- その意味で、**「ベースイメージの最新化が工数的に困難」という理由**で direct usage をサポートしなくなった、という文脈であれば、**distroless に変えると、その問題の深刻さは小さくできる**。  
- 一方で、**CA 証明書**は distroless でも「何らかの形で含める」必要があり、その更新ポリシーをどうするかは別途検討が必要。  
- **「イメージの直接実行もサポートしない」という方針**は、ベースが Alpine か distroless かに関わらず、「公式がそのイメージの継続的な維持を保証しない」という意思表明としては同じにできる。  
- まとめると、**Alpine から distroless に変えると、「パッケージ更新の工数とそれに伴うセキュリティリスク」は軽減され、問題の性質は「維持負荷が大きいフル OS ベース」から「最小ランタイム＋アプリ」に近づく**。

## 4. まとめ

- **ベースイメージを更新しないリスク**: 「セキュリティ」と「CA 証明書の outdated」の両方の仮定は**正しい**。加えて、攻撃面の拡大、フリート全体への影響、修正の複雑さ、上流のサイレントリビルドを取りこぼすリスクなどがある。
- **Terraform**: Alpine ベースで `alpine:latest` を使用。古いアプリバージョン用イメージのベースを定期的に更新している様子は公開情報からは見られず、「おそらく行っていない」という想定でよい。
- **新規リリースと自動化**: 新バージョンリリース時にはその時点の Dockerfile でビルドされるが、既存タグの自動リビルドは説明されていない。Dependabot は「ベースイメージ更新の PR 作成」には使えるが、既存イメージの再ビルド・再プッシュは別設計が必要。ベースの自動更新には破壊的変更のリスクがあるため、CI テスト・ステージング・小さい頻繁な更新などで軽減する。
- **Alpine → distroless**: パッケージ更新の工数とそれに紐づくセキュリティリスクは減り、問題の性質は「フル OS の維持」から「最小ランタイム＋アプリと CA 等の限定的な更新」に近づく。direct usage をサポートしないという方針そのものは、Alpine/distroless どちらでも取り得る。

*このメモは公開情報（ブログ、公式ドキュメント、GitHub Issues/PR、Docker ドキュメント等）に基づく調査結果である。*
